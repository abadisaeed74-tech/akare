# نظرة شاملة على مشروع عقاري (Akare Platform)

## 1) تعريف المشروع
منصة SaaS عربية لإدارة أعمال المكاتب العقارية، تجمع بين إدارة العروض، طلبات العملاء، المتابعات، الفريق، الاشتراكات، والصفحات العامة القابلة للمشاركة.

المنصة تهدف إلى أن تكون أداة تشغيل يومية للمكتب العقاري، وليست مجرد قائمة عقارات:
- إدخال العروض العقارية من نص عربي حر مع تحليل ذكي.
- إدارة طلبات العملاء والعروض المرتبطة بهم.
- مواعيد ومتابعات مركزية للطلبات والعروض.
- توزيع العملاء على الموظفين ومراقبة المسؤوليات.
- صفحات عامة للعقار وللمكتب قابلة للمشاركة.
- إدارة الاشتراكات والخطط والفريق من واجهة واحدة.
- لوحة إدارة للمنصة لمتابعة المكاتب والخطط والاشتراكات.

> ملاحظة: المنصة تعتمد حاليًا على **مفتاح AI موحد** على مستوى النظام، وليس مفاتيح فردية للمستخدمين.

## 2) المعمارية التقنية

### Backend
- **Python + FastAPI**
- **MongoDB** باستخدام Motor/PyMongo
- **Stripe** للفوترة والاشتراكات
- **Gemini API** لتحليل النصوص العقارية
- **Cloudinary** لتخزين الوسائط عند تفعيل إعداداته
- تخزين محلي fallback عبر `/uploads`

### Frontend
- **React + TypeScript + Vite**
- **Ant Design**
- **Axios**
- تصميم RTL عربي بالكامل

### النشر
- **Frontend:** Vercel
- **Backend:** Render
- **Database:** MongoDB Atlas

## 4) الميزات الوظيفية الحالية

### 4.1 إدارة العروض العقارية
- إضافة عرض من نص عربي حر.
- تحليل العرض بالذكاء الاصطناعي واستخراج:
  - نوع العقار
  - المدينة
  - الحي
  - المساحة
  - السعر
  - التفاصيل
  - بيانات المالك/المسوق عند توفرها
- رفع صور متعددة للعقار.
- رفع ملفات مرفقة مثل PDF أو مستندات.
- إضافة رابط خرائط Google للعقار.
- تعديل بيانات العرض يدويًا.
- حذف العرض.
- مشاركة العرض برابط عام.
- توليد **رقم عرض مختصر** `property_code` تلقائيًا لكل عرض.
- عرض رقم العرض داخل لوحة التحكم والصفحة العامة.
- دعم البحث برقم العرض.

### 4.2 روابط فيديوهات العقار
- الفيديوهات أصبحت تعتمد على **روابط خارجية** بدل رفع ملفات فيديو على السيرفر.
- يدعم الحقل روابط:
  - TikTok
  - YouTube / YouTube Shorts
  - Instagram Reels
- في إضافة العرض وتعديله، يتم إدخال كل رابط فيديو في سطر مستقل.
- في الصفحة العامة، يتم تضمين الفيديو داخل الصفحة عبر iframe عندما يكون الرابط قابلًا للتضمين.
- عند وجود رابط غير قابل للتضمين، يظهر كزر فتح الفيديو بدل كسر الواجهة.
- الهدف من هذا القرار:
  - تقليل تكلفة الباندويث والتخزين.
  - دعم سلوك السوق العقاري المحلي المعتمد على TikTok/Reels.
  - إبقاء الزائر داخل صفحة العقار قدر الإمكان.

### 4.3 البحث والتصفح
- بحث نصي كلاسيكي.
- بحث heuristic.
- بحث ذكاء اصطناعي للعقارات.
- دعم البحث بـ **رقم العرض** `property_code`.
- شجرة مدن/أحياء ديناميكية حسب حساب المكتب.
- حذف جماعي حسب المدينة/الحي للمالك.
- عرض إجمالي العروض والمشاهدات والاستفسارات الفعلية في لوحة التحكم.

### 4.4 العملاء والطلبات والعروض
- فصل العملاء إلى:
  - عملاء الطلبات.
  - عملاء العروض.
- وجود ملف عميل موحد يستطيع ربط العميل بأكثر من نوع:
  - `request`
  - `offer`
  - أو الاثنين معًا.
- ملف العميل يعرض:
  - بيانات العميل.
  - الطلبات المرتبطة.
  - العروض المرتبطة.
  - الملاحظات.
  - المواعيد والمتابعات.
- إمكانية إنشاء طلب عميل من نص عربي وتحليله.
- إمكانية ربط العميل بعرض عقاري.
- إمكانية حذف نوع واحد من العميل مع إبقاء ملف العميل إذا كان لديه نوع آخر.
- دعم الملاحظات للطلبات والعروض.
- إصلاح ملاحظات الطلبات بحيث لا تعتمد على endpoint العروض فقط.

### 4.5 الموظف المسؤول عن العميل
- يمكن تعيين موظف مسؤول لكل ملف عميل.
- المالك والمدير فقط يستطيعان تغيير الموظف المسؤول.
- الموظف العادي لا يستطيع تغيير الموظف المسؤول من الواجهة.
- الباكند يرفض أي محاولة من موظف عادي لتعديل:
  - `assigned_user_id`
  - `assigned_user_name`
- في جداول الطلبات والعروض يظهر اختيار الموظف المسؤول للمالك والمدير فقط.
- الموظف العادي يرى اسم المسؤول كقراءة فقط إذا ظهر له.
- في صفحة المواعيد:
  - المالك والمدير يشاهدان اسم الموظف المسؤول مقابل كل موعد.
  - الموظف لا يرى اسمه داخل صفحة المواعيد.

### 4.6 المواعيد والمتابعات
- صفحة مواعيد ومتتابعات مستقلة.
- تجمع المواعيد من:
  - طلبات العملاء.
  - عروض العملاء.
- كل موعد يحتوي على:
  - العميل.
  - نوع المصدر: طلب أو عرض.
  - نوع المتابعة: متابعة أو معاينة.
  - الموعد النهائي.
  - تفاصيل المتابعة.
  - حالة المصدر.
  - الموظف المسؤول للمالك/المدير.
- دعم إكمال الموعد وإغلاق المتابعة.
- دعم تعديل/إعادة جدولة المتابعة.
- دعم فتح ملاحظات الموعد.
- دعم حذف الملاحظات.
- المواعيد الخاصة بالموظف العادي تظهر له فقط إذا كان هو المسؤول عنها.
- المواعيد غير المعينة لا تظهر للموظف العادي.
- المالك والمدير يستطيعان رؤية مواعيد الفريق.

### 4.7 تقويم المواعيد
- تقويم شهري مصغر داخل صفحة المواعيد.
- تمييز أيام وجود مواعيد.
- تمييز اليوم الحالي.
- التنقل بين الأشهر.
- الضغط على يوم في التقويم يفلتر المواعيد لذلك اليوم.
- تبويبات زمنية:
  - الكل
  - اليوم
  - غدًا
  - هذا الأسبوع
  - الأسبوع القادم
  - المتأخرة
- فلترة فعلية حسب:
  - نوع النشاط
  - الحالة
  - مصدر الموعد: طلب أو عرض
  - نطاق التاريخ
- تجميع المواعيد حسب اليوم.
- زر سهم بجانب تاريخ كل يوم لإخفاء/إظهار مواعيد ذلك اليوم.
- قائمة المواعيد نفسها تعمل بسكرول داخلي، بدل نزول الصفحة كاملة.

### 4.8 نظام الحسابات والصلاحيات
- أنواع الحسابات:
  - `owner`: مالك الحساب.
  - `manager`: مدير.
  - `employee`: موظف.
- المدير لديه صلاحيات تشغيلية مثل المالك، لكنه لا يدخل الإعدادات.
- الموظف لديه صلاحيات محددة.
- صلاحيات الموظف:
  - `can_add_property`
  - `can_edit_property`
  - `can_delete_property`
  - `can_manage_files`
- عند إضافة موظف جديد يمكن اختيار نوعه:
  - موظف.
  - مدير.
- المدير يحصل على صلاحيات تشغيلية كاملة تلقائيًا.
- صفحة الإعدادات لا تظهر للمدير أو الموظف من القائمة الجانبية.
- تعديل وحذف العقارات متاح للمالك والمدير، أو للموظف حسب صلاحياته.

### 4.9 الإعدادات والخطط
- إعادة تنظيم صفحة الإعدادات لتتوافق بصريًا مع نمط المنصة.
- تحسين عرض بيانات المكتب.
- تحسين إدارة الشعار والبيانات الرسمية.
- تحسين إدارة الفريق.
- إعادة تصميم صفحة الخطط والاشتراكات بشكل أكثر احترافية.
- عرض حالة الاشتراك والتجربة المجانية.
- أزرار الاشتراك أو إدارة الفوترة.
- دعم فتح Billing Portal من Stripe.

### 4.10 لوحة إدارة المنصة
- صفحة Platform Admin لمتابعة المنصة ككل.
- عرض إحصائيات عامة:
  - عدد المكاتب.
  - الاشتراكات.
  - التجارب المجانية.
  - الموظفين.
  - العقارات.
- عرض قائمة المكاتب.
- فتح تفاصيل مكتب محدد.
- عرض موظفي المكتب.
- عرض بيانات الاشتراك والخطة.
- تنفيذ إجراءات اشتراك للمكتب من لوحة الإدارة.
- الهدف: تمكين مالك المنصة من التحكم التشغيلي في الحسابات والخطط.

### 4.11 الاشتراكات والفوترة
- Stripe Checkout Session.
- Stripe Billing Portal.
- Webhooks لمزامنة الاشتراك.
- Endpoint تأكيد رجوعي لمعالجة تأخير الويب هوك:
  - `POST /billing/confirm-checkout-session`
- خطط قابلة للإدارة من الإعدادات ولوحة الإدارة.

### 4.12 التجربة المجانية
- 30 يوم مجانية.
- مرة واحدة فقط لكل حساب (`trial_used`).
- على خطة `starter` فقط.
- تنتهي تلقائيًا وتحوّل الحساب إلى غير مشترك عند انتهاء المدة.

### 4.13 إظهار/إخفاء العروض حسب الاشتراك
- عند عدم وجود اشتراك نشط:
  - ترجع قوائم العروض والمدن والأحياء والبحث فارغة.
  - ينعكس ذلك على المالك والموظف.
- عند إعادة الاشتراك:
  - تظهر العروض السابقة تلقائيًا بدون إعادة إدخال.

### 4.14 الذكاء الاصطناعي
- مفتاح موحد من `.env`: `GEMINI_API_KEY`.
- إلغاء المفاتيح الفردية من واجهة المستخدم.
- حد يومي لكل شركة: `AI_DAILY_ANALYSIS_LIMIT=30`.
- إعدادات تقليل التكلفة:
  - `GEMINI_MODEL=gemini-2.5-flash-lite`
  - `GEMINI_FALLBACK_MODELS=gemini-2.5-flash`
  - `AI_MODEL_MAX_RETRIES=1`
- fallback آمن عند أعطال Gemini حتى لا تتعطل إضافة العرض.

### 4.15 الصفحات العامة
- صفحة العرض العامة تعرض كل صور العقار.
- تضمين فيديوهات TikTok/YouTube/Instagram داخل الصفحة العامة.
- إخفاء البيانات غير المرغوبة:
  - اسم المالك.
  - رقم المالك.
  - رقم المسوق.
- إخفاء أي معلومة قيمتها `غير مذكور`.
- تحسين رابط الخريطة والتعامل مع الروابط بدون `https`.
- صفحة عامة للمكتب تعرض عروضه.
- بحث داخل صفحة المكتب العامة.

### 4.16 عداد مشاهدات العقار
- عداد مشاهدات للعقار.
- منع تكرار احتساب المشاهدة لنفس المتصفح خلال 24 ساعة.
- التنفيذ يعتمد على Cookie:
  - عند أول زيارة للعقار يتم زيادة العداد.
  - يتم إنشاء cookie باسم مرتبط بالعقار.
  - إذا عاد نفس المتصفح خلال 24 ساعة لا يتم زيادة العداد.
- تم ضبط CORS لدعم الكوكيز بدون استخدام wildcard مع credentials.
- `withCredentials` مفعّل فقط لطلب صفحة العقار العامة، وليس لكل طلبات تسجيل الدخول أو API.

### 4.17 الوسائط والتخزين
- دعم Cloudinary في endpoint الرفع عند تفعيل الإعدادات.
- fallback تلقائي للتخزين المحلي `/uploads` عند عدم تفعيل Cloudinary.
- `resolveMediaUrl` يدعم:
  - الروابط الكاملة.
  - الروابط النسبية.
  - روابط `/uploads`.
  - الروابط القديمة.
- الصور ترفع على المنصة.
- الفيديوهات الجديدة تضاف كروابط خارجية لتقليل تكلفة التخزين والباندويث.

### 4.18 التنقل والقائمة الجانبية
- القائمة الجانبية أصبحت ثابتة في الصفحات الرئيسية.
- قسم المواعيد والمتابعات ظاهر من داخل ملف العميل.
- السايدبار لا يختفي عند التنقل بين صفحات المنصة.
- صفحة المواعيد أصبحت جزءًا أساسيًا من التنقل.

## 5) واجهة المستخدم وتجربة الاستخدام

### 5.1 Landing Page
- عربية بالكامل.
- صفحة الهبوط هي الصفحة الافتراضية.
- قسم الخطط مخفي حاليًا من صفحة الهبوط مؤقتًا.

### 5.2 Dashboard (`/app`)
- أقسام أساسية:
  - نظرة عامة.
  - العقارات.
  - العملاء.
  - المواعيد والمتابعات.
  - الاستفسارات.
  - الإعدادات للمالك فقط.
- شجرة المدن/الأحياء تظهر عند الدخول على قسم العقارات.
- بطاقات العرض ثابتة المقاس مع تحسين الأزرار.
- زر إضافة عرض ضمن السايدبار.
- عرض إجمالي العروض + المشاهدات + الاستفسارات الفعلية.

### 5.3 صفحة العقارات
- عرض العقارات كبطاقات.
- معاينة الصور بدون تكرار.
- فتح تفاصيل العرض داخل Modal.
- مشاركة العقار.
- تعديل أو حذف العقار حسب الصلاحيات.
- إدخال روابط فيديو خارجية عند التعديل.

### 5.4 صفحة العملاء
- تبويبان رئيسيان:
  - الطلبات.
  - العروض.
- جداول مدمجة مع بيانات العميل.
- عرض الموظف المسؤول.
- تعيين المسؤول للمالك والمدير فقط.
- انتقال مباشر لملف العميل.

### 5.5 ملف العميل
- صفحة تفصيلية لكل عميل.
- تعرض الطلبات والعروض والملاحظات والمتابعات.
- يمكن إنشاء متابعة أو موعد من ملف العميل.
- يمكن فتح المواعيد والمتابعات من السايدبار داخل ملف العميل.

### 5.6 صفحة المواعيد
- تصميم تقويم وقائمة احترافي.
- بطاقات إحصائية للمواعيد:
  - المتأخرة.
  - اليوم.
  - هذا الأسبوع.
  - الإجمالي.
- فلترة عملية.
- سكرول داخلي للمواعيد.
- إخفاء/إظهار مواعيد كل يوم.
- عرض الموظف المسؤول للمالك والمدير.

### 5.7 الإعدادات والخطط
- تصميم منظم ومتوافق مع هوية المنصة.
- فصل منطقي بين:
  - بيانات المكتب.
  - الخطة والاشتراك.
  - الفريق والصلاحيات.
- تحسين تصميم الخطط بصريًا.

### 5.8 الثيم
- الوضع الليلي ملغى حاليًا.
- اعتماد هوية بصرية فاتحة:
  - أخضر.
  - رمادي.
  - أبيض.
- الواجهة RTL وموجهة للمستخدم العربي.

## 6) API Endpoints الرئيسية

### Auth
- `POST /auth/register`
- `POST /auth/login`
- `GET /me`
- `PUT /me/display-name`

### Properties (Private)
- `POST /properties`
- `GET /properties`
- `PUT /properties/{id}`
- `DELETE /properties/id/{id}`
- `GET /cities`
- `GET /neighborhoods`
- `GET /search`
- `GET /ai-search`
- `POST /upload`

### Public
- `GET /public/properties/{id}`
- `POST /public/properties/{id}/inquiries`
- `GET /public/companies/{owner_id}`
- `GET /public/companies/{owner_id}/properties`
- `GET /public/companies/{owner_id}/ai-search`

### Clients / Requests
- `POST /clients`
- `GET /clients`
- `PUT /clients/{client_id}`
- `DELETE /clients/{client_id}`
- `GET /clients/{client_id}/notes`
- `POST /clients/{client_id}/notes`
- `DELETE /clients/{client_id}/notes/{note_id}`
- `GET /clients/{request_id}/matches`

### Client Offers
- `POST /clients/offers`
- `GET /clients/offers`
- `GET /clients/offers/{offer_id}`

## 9) ملاحظات تشغيل/إطلاق
- ملف `backend/.env` محلي فقط ولا يجب رفعه.
- في Render يجب ضبط env vars يدويًا.
- إذا كان دومين الواجهة مختلفًا عن `FRONTEND_BASE_URL` يجب إضافته إلى `CORS_ORIGINS`.
- لا تستخدم `allow_origins=["*"]` مع `allow_credentials=True`.
- يجب تثبيت تبعيات الباكند بعد أي إضافة جديدة في `requirements.txt`.
- `frontend/vercel.json` مهم لتفادي 404 على مسارات SPA.
- يفضل تدوير مفاتيح Stripe/Gemini/Cloudinary دوريًا.
- يفضل إبقاء الفيديوهات كرابط خارجي لتفادي تكلفة egress والتخزين.
